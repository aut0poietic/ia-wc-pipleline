Unfortunate Misadventures inWeb Components

—or How I learned to stop worrying about and love the Shadow DOM.

01 What is this about?

So apparently, there are these things called Web Components.

Who knew…

What with all the React's, Vue's and Svelte's (oh, my) I think I could be forgiven for not paying attention, despite the apparent controversies.

This year, I began planning a fairly complex project: A special-purpose "static site generator" with an interactive visual editor. The editor was designed to create something not unlike a visual novel — heavy on the animation and interactivity; dynamic, branching story lines; a pluggable system for adding new elements and templates. Oh, and it had to communicate via a standardized specification.

ReactJS was the first and obvious choice for creating this. However I quickly ran into issues. I found myself falling down a rabbit hole of react running react running react… When my experimental projects started weighing in at multiple MB, I realized React was not the way forward for the static site. Many of my target audience had project size quotas of under 4MB.

I landed on VanillaJS for it's speed and size.

I entertained the others briefly but landed on VanillaJS for the static site and React for the editor itself. But having experienced the speed of development that React and similar tools allow, I had a hard time stomaching the idea of developing the entire project without some form of reactive framework. I started looking around for smaller so-called "micro-frameworks" to see if I could ease the transition and stumbled onto LitElement and Web Components.

The idea was that I could build pre-built web components as plugins that are already in their final state -- all the editor has to do is provide the data/children. There are some fairly significant technical hurdles (like teaching the editor what properties & children are valid and what values they accept), but if React can treat a Web Component tag like any other HTML tag, the entire project just became significantly easier to build.

The problem is that I came to this from LitElement's (and to a lesser extent, React's) perspective. I know nothing about Web Components as a technology, how they work, how they live nor whether they're viable. There are a lot of opinions on what components are and what they're useful for or whether they're useful at all.

Blindly accepting the opinions of the peanut gallery seems like a bad idea. What's a developer to do?

02 Throwing it All Away

This repo is a series of simple experiments for me to explore developing web components to solve the needs of the project I described above.

I assume nothing, throwing out every shred of conventional wisdom, every so-called best practice and all of the words of any venerated expert.

I am intentionally re-treading ground, often doing things that are completely, mind-bogglingly moronic.

This is all so I can produce components that:

  • Generate self-contained components that can be used anywhere with no dependencies that are extremely small.
  • Identify the build tooling to allow for one-component-per-js-file, with multiple components living in the same project.
  • Refine the tooling to allow for writing CSS, JavaScript and HTML in their own files (no CSS/HTML-in-JS) that are bundled into a single component file with no overhead.
  • Standardize the tooling and configuration so it can live happily within another framework's build config transparently.
  • Utilize the component as content within another application built with React, Vue or Svelte as if it were any other HTML tag. This requires that the parent application manipulate the attributes and children of the components naturally.
  • Augment the components with compatible technologies: Workers, Assembly, whatever.
  • Fail, publicly.
  • Have fun building silly experiments I don't ordinarily get to do at the Day Job™.

As I've progressed in this, I've divided the experiments up into sections:

  • Build & Tooling
    —Experiments trying to muddle my way towards the build-related goals.
  • Components
    —Experiments in building reusable, self-contained components.
  • Integration
    —Create React/Vue/Svelte applications that consume the base components.

03 Build & Tooling

Simple experiments attempting to achieve a solid build process for web components that are compatible with React/Preact, their particular build configurations and my personal development quirks.

03:01 Webpack Default

Example Source

This is an intentionally stupid attempt at using Webpack to build a web component with no specialized tooling or build process. The only non-standard webpack bit is the css-to-string-loader. As you'd expect with an attempt named "stupid", it didn't go so well.

File Weight
Source JS 448 B
Source CSS 326 B
Production Bundle 22 KB

The primary issue is size. LitElement is a 3K library — if you don't count the 17 other files that are included via, well, include. In the end, LitElement weighs in at around 43K, all files included. Once Webpack adds in it's overhead, then minifies the whole thing, the resulting file is 22K.

On top of this are issues with the nature of webpack and LitElement. Both are much happier using css-in-js which was completely counter to the goal. Webpack dumps the entire css file in a variable, which has to be assigned to a class property on the component. In order to make that kludge work, the entire CSS file has to be passed through LitHTML's unsafeCSS method. The resulting FOUC was extreme.

Result: Failed

03:02 Webpack Reduced TC

Example Source

This example is a reduced text case version of the previous Webpack build process. The idea was to strip out anything that could be inflating the file size and see what the bundled code would look like.

In light of that, I removed LitElement, building the component natively and went with CSS & HTML in the JavaScript (even though it's counter to my goals and the way I enjoy writing code). I took the time to hand-minify the HTML & CSS as Webpack can't minify the content of strings.

File Weight
Source JS < 1 KB
Production Bundle 2 KB

The result was worth one of those face-palm moments.

Obviously, the reduced test case worked, producing a bundle file was 2K. If you think of the result compared to the LitElement's 22K, it'd be understandable if you saw this as a successful test.

But seeing as the original file was just slightly less than 1K, a final bundle of 2K is not an acceptable result. Minification should not increase the size of the bundle.

It's also throwing out all the goals of the project itself.

Result: Failed

03:03 RollUp Reduced TC

Example Source

This example is the same reduced text case version that substitutes RollUp over webpack. I noticed that LitElement uses RollUp in most of it's examples. In any project, it's hard to know if the choice was one based on preference, politics or if there's a valid reason for the choice.

Again, I removed LitElement, building the component natively and used CSS & HTML in the JavaScript with hand-minified the HTML/ CSS.

File Weight
Source JS 994 B
Production Bundle 848 B

This reduced test case was more what I expected from the previous attempt.

The final bundle is smaller but not significantly so, which is to be expected from this test — the majority of the ‘weight’ of the file is located in the CSS & HTML and there should be some minification gains beyond what I could produce by hand with a purpose-built build plugin.

Result: Progress

03:04 RollUp External CSS

Example Source

Once again, this is the same component in previous examples, however this time around I moved the CSS to it's own file and removed my hand-minification. The HTML is a simple template literal. The CSS is loaded using the usual import statement and the variable the style were loaded into placed into the template placeholder.

File Weight
JS 754 B
CSS 307 B
Bundle 892 B

The result worked so well, I went ahead and included PostCSS with autoprefixer and flex-bugs-fixes, which is why the bundle file is slightly larger than the previous experiment.

Even better, the final, minified code is surprisingly clean. Unlike Webpack, RollUp did away with the import for the CSS and injected it directly into the template string after running it through PostCSS.

In the end, this is a fairly usable configuration and shows the way for further development.

Result: Success

The result here is fairly lack-luster. Like Webpack, Rollup bundled the entire LitElement package into the final file. Worse, Rollup refused to inline the CSS as it had in the native example. The build failed until I wrapped the imported CSS with unsafeCSS.

It's a case of "one-step-forward, one-step-back." In the end, this has soured me on the use of LitElement, which is a shame.

I'd say this is the first fully-successful attempt to create something I could use for a production project. Still, it does precisely what I wanted with external CSS, and each file in it's own directory, with each as it's own entry point. This produces a file that can be used anywhere.

04 Components

These experiments are all about attempting to build components using various techniques. Most of these examples will refine the build configurations in ways tailored to the component's needs.

Unlike the components in the Build section, any of these successful components should be production quality, with usable accessibility models, fallback styles (when appropriate) and minimized files suitable for use in any project.

All of the components are MIT licensed. If you like them, want to use or refine them, please feel free. And for anyone asking, the ia- prefix is for Irresponsible Art, in other words, me.

04:01 Cross Carousel

Example Source

I liked the idea of an image carousel that "split" the image when animating between images. I had in mind an transition animation that was like an arc welder splitting the image in half, with sparks and everything. I let the animation go in this version, as the particle effect was fairly heavy, but the final result is still pretty useful.

File Weight
Carousel JS 6 KB
Carousel CSS 2 KB
Carousel HTML 845 B
Slide JS 2 KB
Slide CSS 2 KB
Slide HTML 235 B
Production Bundle 10 KB

The carousel is actually made up of two different components, both bundled into the same file as they cannot be used separately. I managed to refine the build process so that I could write the JavaScript, CSS & HTML in separate files.

I built in the ability to turn off both the next/previous navigation and the "bullets" that appear below the images to control and indicate the current slide. Because of this I invested some time in the accessibility of the component, reviewing the ARIA Authoring Practices and implementing a dual-model for the component.

Without bullets, the carousel uses the group role to define the slides behavior, and the next/previous buttons control the component. When the bullets are turned on, the accessibility model shifts to that of a Tab/TabPanel with supplemental navigation.

I ensured that the external attributes navigation, buttons, and index can all be manipulated using external Javascript. This should allow another framework to control and potentially build a carousel as if it were HTML. I also took the time to build a fallback for if JavaScript is turned off or if the component itself is not supported. But this revealed a problem…

Web Components are JavaScript; all of the structure and styles are implemented in JavaScript and if JavaScript is not enabled, there's no way to control the appearance of the component when it fails from within the component. So the fallback can't be bundled with the component itself.

The fallback for a component can't be bundled within the component itself.

At this point, I'm implementing the fallback in the hosting HTML, using ia-carousel:not(:defined) styles. The pattern I chose reveals only the image for the first slide, and attribute content to handle the title of the slide. The fallback style sets the image for the slide as the background image, but if I were to redesign the component, I'd likely use a slide:before{ content: attr(src); } instead.

I'm not sure how to handle the fallback, short of providing a fallback.css file that accompanies the component. It's counter to the single 'universal' file for each component, but it at least would help end-users using the component have a useful, default fallback strategy.
